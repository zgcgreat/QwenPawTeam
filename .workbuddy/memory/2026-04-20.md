## 修复 PluginProvider 在登录页触发 401

**问题**：远端新功能 `PluginProvider`（插件系统前端）放在 `App` 顶层包裹整个应用，mount 时立即 `GET /api/plugins`。用户在登录页时未认证，请求返回 401。

**根因**：`PluginProvider` 位于 `ActiveAuthGuard` 之外，登录页也在其包裹范围内。

**修复**：将 `PluginProvider` 从 `App` 顶层移到 `ActiveAuthGuard` 内部，包裹 `MainLayout`。只有认证通过后才 mount `PluginProvider`，触发插件加载。

改动文件：`console/src/App.tsx`（仅调整 JSX 结构，零侵入多租户逻辑）

## 修复 token_usage monkey-patch 遗漏（agent_stats 数据泄露）

**问题**：新建租户的智能体统计 API `/api/agent-stats` 返回的 token 数据（prompt_tokens/completion_tokens/llm_calls）非零，泄露了其他租户的数据。chats 和 messages 为零（隔离正确）。

**根因**：`agent_stats/service.py` 第 17 行 `from ..token_usage import get_token_usage_manager` 是模块加载时的本地绑定。`token_usage_extension.py` 只 patch 了 4 个模块（manager、package、model_wrapper、router），遗漏了 `agent_stats.service`。同理 `agents.tools.get_token_usage` 也被遗漏。

**修复**：在 `token_usage_extension.py` 的 `patch_token_usage_manager()` 和 `unpatch_token_usage_manager()` 中增加 Level 5 (`agent_stats.service`) 和 Level 6 (`agents.tools.get_token_usage`) 的 patch。

**教训**：Python 的 `from X import Y` 创建的是本地绑定，monkey-patch `X.Y` 不会影响已加载模块的 `Y`。每次远端新增引用 `get_token_usage_manager` 的模块时，都需要同步更新 patch 列表。

## 修复 token_usage 和 agent_stats 按 agent 级别隔离

**问题**：`/api/token-usage` 和 `/api/agent-stats` 的 token 数据没有按智能体（agent）级别隔离，仅按租户级别隔离。同一租户下多个 agent 的 token 使用量混在一起。

**根因分析**：
1. 原始代码中 `TokenUsageManager` 是全局单例，路径为 `WORKING_DIR / token_usage.json`，所有 agent 共享——**原始代码就不按 agent 隔离**
2. 多租户 `token_usage_extension.py` 改为按租户隔离（`{SECRET_DIR}/tenants/{tenant_id}/token_usage.json`），但粒度不够细，应为按 (tenant, agent) 隔离
3. `/api/token-usage` 路由没有像 `/api/agent-stats` 那样通过 `get_agent_for_request(request)` 确定当前 agent
4. `AuthMiddleware` 只设置 `set_current_tenant_id` 和 `set_current_workspace_dir`，没有设置 `set_current_agent_id`

**修复**（两处改动，零侵入原始代码）：
1. `token_usage_extension.py`：将隔离粒度从 `{tenant_id}` 改为 `{tenant_id}/{agent_id}`，路径变为 `{SECRET_DIR}/tenants/{tenant_id}/{agent_id}/token_usage.json`。`_resolve_agent_id()` 通过 `get_current_agent_id()` 获取当前 agent
2. `auth_extension.py`：在 `AuthMiddleware.dispatch()` 中增加 `set_current_agent_id(agent_id)` 调用，从 `X-Agent-Id` header 获取 agent_id 并传播到 ContextVar

**改动文件**：
- `src/qwenpaw_plugins/multi_tenant/token_usage_extension.py` — 重写为 (tenant, agent) 粒度
- `src/qwenpaw_plugins/multi_tenant/auth_extension.py` — AuthMiddleware 中增加 agent_id ContextVar 传播

## 修复备份功能按租户隔离

**问题**：备份功能（创建/列表/详情/删除/导出/导入/恢复）完全没有按租户隔离。所有操作使用全局路径常量（`BACKUP_DIR`、`WORKING_DIR`、`SECRET_DIR`、`CONFIG_FILE`），任何租户可以看到/删除/导出其他租户的备份，恢复备份到全局目录覆盖其他租户数据。

**根因**：7 个备份子模块通过 `from ...constant import X` 绑定了全局路径常量，没有租户感知。

**v1 修复（失败）**：路由层 handler 包装 + swap/restore 路径常量。**失败原因**：FastAPI `include_router` 会复制路由对象，`route.endpoint = _wrap_handler(route.endpoint)` 修改的是子路由的 endpoint，不影响已注册到 app 的路由。

**v2 修复（当前）**：用 `TenantAwarePath` 动态代理替换模块级路径常量，与 `console_extension.TenantAwareLogPath` 同一策略。每次属性访问时动态读取 `get_current_tenant_id()` ContextVar，天然线程安全，无需 swap/restore/asyncio.Lock。

**租户备份目录布局**：`{BACKUP_DIR}/tenants/{tenant_id}/{backup_id}.zip`

**patch 的 7 个模块**：`backup._utils.constants`、`backup._ops.storage`、`backup._ops.create`、`backup._ops.create_helpers`、`backup._ops.restore`、`backup._ops.restore_helpers`、`app.routers.backup`

**关键教训**：FastAPI `include_router` 会复制路由对象，不能通过修改子路由的 `route.endpoint` 来影响 app 中已注册的路由。应使用模块属性替换（动态代理/函数替换）策略。
